## なぜ存在するか

Koutaro は2つの nb ノートブックに markdown ノートを蓄積している：

- **home** (`~/.nb/home/knowledge/`, 400+本) — 個人の技術学習・設計判断・dotfiles・キャリア・求人リサーチ
- **work** (`~/.nb/work/knowledge/`, 170+本) — 業務案件の知見（8122 / cameraman / leader、API 認証認可・IDOR 監査、配信基盤、Terraform 等）

どちらも Cloudflare AI Search でインデックスされ MCP ツールとして公開されている。しかし放っておくと Claude はこの蓄積を忘れて Web 検索や一般知識で答えてしまい、**せっかくのナレッジが活用されない**。このスキルはその防止策であり、かつ「どの検索手段が最適か」を判断するためのもの。

## どのノートブックを検索するか

質問の文脈で振り分ける。**トピックが最も強いシグナル**で、PC 種別は補助。

| 手がかり | 検索先 |
|---|---|
| 業務・案件名（8122 / cameraman / leader / sen）、IDOR・認可監査、配信基盤、業務 Terraform、社内固有の話 | **work** |
| 個人技術学習、dotfiles、Nix、キャリア・転職・Findy・ファンテック、汎用フレームワーク調査 | **home** |
| どちらとも判断つかない | **両方**（home と work を並行で投げる） |

- `~/.is_work_pc` が存在する＝業務マシンなら、曖昧時のデフォルトを **work** 寄りに。無ければ **home** 寄り。
- 両ツール（`mcp__home-memo__search_knowledge` / `mcp__work-memo__search_knowledge`）はどちらの PC でも利用可能。PC 種別はあくまで「どちらを先に/優先で引くか」のヒント。
- **迷ったら両方投げてコスト惜しまない**。取りこぼしの方が痛い。

## MCP（セマンティック検索）と grep の使い分け

これは実測で検証した使い分け。両者は上位互換ではなく**相補的**。

### MCP セマンティック検索が勝つ場面 → まずこちら

- **言語が跨ぐとき**（最重要）。英語クエリで日本語本文を探す／その逆。grep は英語 "guard" で日本語「ガード」に当たらず、**一番大事なノートを丸ごと取りこぼす**。MCP は意味で当てる。
- **言い換え・概念検索**。「あの概念どこかにまとめたっけ」「言い回しうろ覚え」。grep は順不同の大量ヒットを吐くだけだが、MCP は関連度スコア＋抜粋で上位提示する。
- **別マシンから**。ローカルにノートを clone していなくても MCP はリモートで引ける。
- 「とりあえず当たりを付けたい」初手。

### grep（ローカル全文検索）が勝つ場面

- **完全一致・網羅性**。識別子・ファイルパス・エラー文字列・関数名（`photographer_id`、`throttle:5,1` 等）の一撃検索。レイテンシ0・無料・インデックス遅延なし・常に最新。
- **取りこぼし厳禁のとき**。MCP の top-K は関連ノートを**落とすことがある**（実測で確認）。grep は該当を**全部**返す（再現率100%）。
- 対象: `~/.nb/home/knowledge/` と `~/.nb/work/knowledge/` の `*.md`。

### 推奨フロー：MCP で当たり → grep で確定

1. **MCP でヒット候補とファイル名を掴む**（言語跨ぎ・概念に強い）
2. 網羅性が要る・MCP の結果が薄い/怪しいときは、得たキーワードで **grep して該当ファイルを exhaustive に洗う**
3. 必要なら該当ファイルを Read して原典確認

```
ユーザー: 「cameraman の認可監査でカメラマンのガード名どうなってたっけ」
→ MCP work: "photographer guard 命名" でヒット候補を掴む
→ 不足なら grep: rg -l "guard|ガード|photographer_uploader" ~/.nb/work/knowledge/
```

## 使い方（MCP 検索の実務）

### 1. クエリを組み立てる

ユーザーの質問の **核となるキーワード** を抽出する。長文をそのまま投げない（semantic 検索の精度が落ちる）。

**言語選択ルール:**
- ユーザー発話と同じ言語を主軸に（日本語ノートは日本語クエリ、英語ノートは英語クエリの方がヒット率が高い）
- 技術用語は英語のまま (`middleware`, `routing`, `WebSocket`)
- 固有概念・日本語特有の文脈（ファンテック、Findy、企業名、案件名）は日本語のまま
- 日英混在クエリも有効（例: `"Hono middleware 認証"`）
- **言語跨ぎで取りこぼしが怖いときは、和英両方のクエリを投げる**（grep では拾えない領域なので MCP の出番）

```
「TanStack Router のファイルベースルーティングってどう書くんやっけ？」 → "TanStack Router file-based routing"（home）
「8122 のカメラマン API の認可どうなってた？」 → "cameraman photographer API 認可 IDOR"（work）
「前にまとめたファンテック企業のリスト見せて」 → "ファンテック 企業 リスト"（home）
```

### 2. ツールを呼ぶ

```
mcp__home-memo__search_knowledge(query="...", max_results=5)
mcp__work-memo__search_knowledge(query="...", max_results=5)
```

**`max_results` の段階的拡張ルール:**
- 1 回目: `5`（デフォルト）
- ヒット0 か明らかにズレている場合のみ `10` で再検索
- それでも薄ければ `20` で打ち止めし、grep か Web フォールバックへ

やみくもに最初から `20` を投げない。ノイズが増えて semantic 検索の精度が落ちる。

### 3. 結果を回答に組み込む

**ヒットありの場合:**
- チャンク内容を要約しつつ、**ファイル名を `knowledge/xxx.md` の相対パス形式**で引用する（原典をたどれるように）。どちらのノートブックか分かるよう home/work も添える
- 複数ヒット時の優先順位: ツールが返した **semantic スコア順を主軸**に、年月が分かるノートは **新しい日付を優先**（古い情報を最新と誤認させない）
- ヒット内容と一般知識を**混ぜて答えてよい**
- ノートと一般知識が食い違うときは **ノート優先**かつ食い違いを明示（ノートが古い可能性もあるので年月を添える）

**ヒットゼロ／薄い場合の手順:**
1. **回答冒頭で**「ノートには見つからなかった」と明示する（透明性のため、結果に紛れ込ませない）
2. 別キーワードで **1回だけ** 再クエリ。軸の振り方: 同義語／英訳↔和訳／関連語に拡張／具体化（`auth`→`Bearer 認証`）。複数軸を同時に変えても可（元クエリと意味的につながる範囲で）
3. **MCP が空でも grep を試す価値がある**（完全一致・別表記を拾える）。逆に grep が空でも MCP を試す価値がある（言い換え・概念）。両者は取りこぼす領域が違う
4. それでも空なら一般知識・Web 検索にフォールバック
5. 新規の学びがあれば「ノートに残しますか？」と促すか、`nb-knowledge` 経由で書き残す

### 4. 関連スキルとの連携

- **nb-knowledge**: 検索結果が古い／新しい学びを得たら nb-knowledge が発火して新規ノートを書く（こちらと書き手で対）
- **job-hunter**: 求人系の検索ではこちらと併用。過去リサーチがあれば再利用して重複を避ける

## 重要

- **「Web 検索の前にまず knowledges」** を癖にする。Web より速いし、ユーザー固有の文脈が乗っている
- 検索結果が空でも**呼んだことを隠さない**（「ノートには見つからなかったので一般知識で答える」と明示）。何をしたか説明することで信頼できる
- クエリは**短く・キーワード中心**に
- 同一セッションで同じトピックを再度問われたら、2度目以降の検索は省略してよい（直前コンテキストに結果が残っている）。**ただし「別の角度」「もっと深く」と新しい切り口を要求された場合は再検索**
- **どのノートブックを引くか迷ったら両方**。**どの手段で引くか迷ったら MCP→grep の順**。取りこぼしが一番の損失

### 境界条件（迷いやすいケース）

- **「最新バージョンの skip」**: 「**今の**最新バージョン」「**現在の**〜」のように現在値・最新値を問う場合のみ skip。「うちのプロジェクトで固定してた〜のバージョン覚えてる？」「前に決めた依存バージョン」のように **過去の自分の決定を参照する場合は検索する**
- **「コードベース内部質問」**: 「このリポジトリの `src/foo.ts` を読んで」のような直接読解は skip。ただし「**前に**このリポジトリで〜のパターン使ってたやろ？」のような **過去判断の参照は検索する**
- **「業務 vs 個人の振り分け」**: 案件名・社内固有語が出たら work、汎用技術・キャリア・dotfiles なら home、判断つかなければ両方
